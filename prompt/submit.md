# 代码提交指南

## 1. 提交前准备

### 1.1 环境检查
- [ ] 代码已通过测试：`pytest`
- [ ] 代码符合PEP 8规范
- [ ] 所有修改已完成，无未完成的工作
- [ ] 工作目录干净，无临时文件或未跟踪文件

### 1.2 分支检查
- [ ] 确认当前分支是正确的开发分支
- [ ] 确保分支是最新的：`git pull`

## 2. 提交流程

### 2.1 检查修改状态
```bash
# 查看当前项目状态
git status

# 查看具体的代码变更内容
git diff

# 列出所有变更的文件路径
git diff --name-only
```

### 2.2 暂存文件
```bash
# 暂存所有修改的文件
git add .

# 暂存特定文件
git add path/to/file

# 检查暂存状态
git status
```

### 2.3 提交修改
```bash
# 提交修改到本地仓库
git commit -m "[类型] 简短描述"

# 提交修改并添加详细说明
git commit -m "[类型] 简短描述" -m "详细说明"

# 修改最后一次提交信息
git commit --amend
```

### 2.4 推送修改
```bash
# 推送修改到远程仓库
git push

# 推送修改到特定分支
git push origin branch-name

# 第一次推送新分支
git push -u origin branch-name
```

## 3. 提交信息规范

### 3.1 提交信息格式
```
[类型] 简短描述

详细说明（可选）

影响范围：模块1, 模块2
```

### 3.2 提交类型
- **feat**：新功能
- **fix**：bug修复
- **docs**：文档更新
- **style**：代码风格修改
- **refactor**：代码重构
- **test**：测试用例修改
- **chore**：构建/工具修改

### 3.3 提交信息示例
```
feat: 添加文本连续判断提示词优化

- 修改translator.py中的语义分析提示词
- 添加明确的标题识别规则
- 提供详细的标题示例和非标题示例

影响范围：translator, test_semantic_merge_extended
```

## 4. 分支管理

### 4.1 分支类型
- **main**：主分支，仅用于发布稳定版本
- **develop**：开发分支，整合各功能分支
- **feature/xxx**：功能分支，用于开发新功能
- **bugfix/xxx**：bug修复分支
- **release/xxx**：发布分支，用于准备发布

### 4.2 分支操作
```bash
# 创建新分支
git checkout -b feature/xxx

# 切换分支
git checkout branch-name

# 合并分支
git merge branch-name

# 删除分支
git branch -d branch-name
```

## 5. 常见问题处理

### 5.1 冲突处理
1. 识别冲突文件
2. 手动解决冲突
3. 标记冲突已解决：`git add conflicted-file`
4. 完成提交：`git commit`

### 5.2 撤销修改
```bash
# 撤销工作区修改
git checkout -- file

# 撤销暂存区修改
git reset HEAD file

# 撤销最后一次提交（保留修改）
git reset HEAD~1

# 撤销最后一次提交（丢弃修改）
git reset --hard HEAD~1
```

### 5.3 其他常见问题
- **推送被拒绝**：先拉取最新代码，解决冲突后再推送
- **忘记添加文件**：暂存并提交：`git add file && git commit --amend`
- **提交信息错误**：修改提交信息：`git commit --amend`

## 6. 提交最佳实践

### 6.1 提交频率
- 每完成一个小功能或修复一个bug就提交一次
- 不要长时间不提交代码
- 提交应保持原子性，每次提交只包含一个逻辑变更

### 6.2 提交内容
- 只提交源代码和必要的配置文件
- 不要提交临时文件、日志文件或构建产物
- 确保 `.gitignore` 文件正确配置

### 6.3 团队协作
- 定期拉取最新代码，避免冲突
- 及时推送本地修改，确保团队成员能够看到
- 大的变更前先与团队成员沟通

## 7. 验证清单

### 7.1 提交前验证
- [ ] 代码已通过测试
- [ ] 代码符合PEP 8规范
- [ ] 所有修改已完成
- [ ] 工作目录干净
- [ ] 分支是最新的

### 7.2 提交过程验证
- [ ] 正确暂存了所有修改的文件
- [ ] 提交信息格式正确
- [ ] 提交信息描述清晰完整
- [ ] 成功推送到远程仓库

### 7.3 提交后验证
- [ ] 远程仓库已更新
- [ ] 构建/CI已通过（如需）
- [ ] 通知团队成员（如需）

## 8. 注意事项

- **确保数据安全**：提交前备份重要数据，以防意外情况发生
- **编写有意义的提交信息**：以便其他开发人员理解所做的修改
- **遵循分支管理规范**：确保在正确的分支上工作
- **及时解决冲突**：遇到冲突时，及时与团队成员沟通解决
- **定期清理分支**：删除不再需要的分支，保持仓库整洁
