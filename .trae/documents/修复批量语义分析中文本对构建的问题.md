## 根本原因分析

**问题现象**：从日志中可以看到，所有批量分析的文本对中，块1的文本都是相同的，都是"This "Think, Act, Observe" cycle continues - manag..."，而块2的文本则不同。

**根本原因**：在`merge_semantic_blocks_with_llm`函数中，文本对的收集逻辑存在问题：

1. **文本对收集阶段**：在收集批量处理的文本对时（第671-684行），代码始终使用初始的`current_merged.block_text`作为所有文本对的前一个块文本
2. **结果处理阶段**：虽然在处理批量分析结果时会更新`current_merged`，但这是在所有文本对都收集完成并分析后才进行的

**具体问题代码**：
```python
while i < total_blocks and len(text_pairs) < batch_size:
    curr_block = text_blocks[i]
    prev_text = current_merged.block_text  # 始终使用相同的prev_text
    curr_text = curr_block.block_text
    
    # 检查特殊情况
    if is_image_caption(prev_text) or is_new_paragraph_start(curr_text):
        break
    
    text_pairs.append((prev_text, curr_text))
    batch_blocks.append(curr_block)
    i += 1
```

## 修复方案

### 方案选择

**方案1**：动态更新收集逻辑 - 在收集文本对的同时更新`current_merged`
- 优点：逻辑直观
- 缺点：破坏批量分析的目的，因为需要在分析前收集所有文本对

**方案2**：累积处理结果 - 收集文本对时使用当前状态，处理结果时累积更新
- 优点：保持批量分析的效率
- 缺点：需要修改结果处理逻辑

**方案3**：模拟累积更新 - 在收集文本对时模拟累积更新过程，生成正确的文本对序列
- 优点：保持批量分析效率，同时生成正确的文本对
- 缺点：实现复杂度较高

**选择方案3**：因为它既能保持批量分析的效率（减少API调用），又能生成正确的文本对序列，确保语义分析的准确性。

### 修复步骤

1. **修改文本对收集逻辑**：
   - 在收集文本对时，创建一个临时的`temp_merged`变量来模拟累积更新过程
   - 对于每个文本对，使用`temp_merged.block_text`作为prev_text
   - 如果当前文本对应该合并（基于临时判断），更新`temp_merged`

2. **修改批量分析结果处理逻辑**：
   - 使用实际的批量分析结果来更新`current_merged`
   - 确保每个后续的分析结果都基于之前的更新

3. **添加详细日志**：
   - 在文本对收集和处理过程中添加详细日志，以便于调试和验证

4. **测试验证**：
   - 运行测试套件验证修复是否有效
   - 检查日志确保文本对的构建正确

### 预期效果

修复后，日志中的文本对应该显示为：
1. 文本对 1：块1="文本1"，块2="文本2"
2. 文本对 2：块1="合并后的文本1+文本2"（如果前一个分析结果为合并），块2="文本3"
3. 文本对 3：块1="更新后的合并文本"，块2="文本4"
4. 以此类推

这样才能确保批量语义分析的正确性，从而提高翻译结果的质量。
