# 优化PDF提取和表格翻译流程

## 问题分析

用户明确指出，在`PdfExtractor.extract`方法中，表格提取已经按页面范围进行了处理，所以不需要在`translate_tables`函数中再进行页面范围的判断。

当前实现中存在以下问题：

1. **冗余数据传递**：`extract_pdf_content`函数返回整个`extracted_content`对象，但除了表格信息外，其他信息已经被处理或存储在`all_blocks`中
2. **冗余页面范围检查**：`translate_tables`函数中对表格进行页面范围过滤，但表格已经在提取阶段按页面范围过滤过了
3. **内存使用**：保存整个`extracted_content`对象会占用不必要的内存

## 优化方案

### 方案：完整优化PDF提取和表格翻译流程

1. **修改** **`extract_pdf_content`** **函数**：
   * 不再返回`extracted_content`对象
   * 改为返回`(tables, extracted_images, target_pages, all_blocks)`，其中`tables`是从`extracted_content.tables`获取的

2. **修改** **`translate_tables`** **函数**：
   * 移除`extracted_content`和`target_pages`参数
   * 直接接受`tables`参数
   * 移除页面范围检查逻辑，因为表格已经按页面范围提取

3. **修改** **`process_translation`** **函数**：
   * 更新`extract_pdf_content`的调用和返回值处理
   * 直接将返回的`tables`传递给`translate_tables`
   * 移除对`extracted_content`的引用

## 具体修改步骤

1. **修改** **`extract_pdf_content`** **函数**：
   * 更新函数签名和返回值类型
   * 从`extracted_content`中提取`tables`信息
   * 返回`(tables, extracted_images, target_pages, all_blocks)`

2. **修改** **`translate_tables`** **函数**：
   * 更新函数签名，移除`extracted_content`和`target_pages`参数
   * 直接使用传入的`tables`参数
   * 移除页面范围检查逻辑

3. **修改** **`process_translation`** **函数**：
   * 更新`extract_pdf_content`的调用，接收新的返回值
   * 直接将返回的`tables`传递给`translate_tables`
   * 移除对`extracted_content`的引用

## 预期效果

1. **减少内存使用**：不再需要保存整个`extracted_content`对象
2. **提高代码清晰度**：函数职责更加明确，数据流转更加直观
3. **简化代码逻辑**：移除不必要的参数传递和页面范围检查
4. **保持功能完整性**：优化不会影响任何现有功能

## 风险评估

* **低风险**：修改仅涉及内部实现，不影响外部API
* **兼容性**：需要更新相关测试用例，但不会影响其他模块
* **测试覆盖**：所有测试用例应该继续通过，确保功能完整性
