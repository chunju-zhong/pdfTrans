## 二、多线程并行化方案分析

### 2.1 如果对每对文本并行调用API (方案A)

**并行方式：** 同时发起20次 API 调用，每次判断1对文本

```
线程1: 文本对1 → LLM1 → 结果1
线程2: 文本对2 → LLM2 → 结果2
线程3: 文本对3 → LLM3 → 结果3
... (共20个并发线程)
```

**问题分析：**

| 问题类型 | 具体描述 | 影响程度 |
|---------|---------|---------|
| **上下文丢失** | 并行调用时，第N对分析时无法获知前N-1对的实际合并结果 | ⚠️ 严重 |
| **输入文本不一致** | 如果使用 `current_merged.block_text` 作为块1输入，由于并行执行，无法确定哪个结果先返回 | ⚠️ 严重 |
| **API限流风险** | 20个并发API调用可能触发服务商的限流机制 | ⚠️ 中等 |

**为什么会有上下文丢失：**

假设有文本块 A, B, C, D，正确的顺序执行逻辑：

```
1. 判断 A+B 是否合并 → current_merged = A (假设不合并)
2. 判断 B+C 是否合并 → current_merged = B (假设合并) → current_merged = B+C
3. 判断 C+D 是否合并 → current_merged = B+C (假设不合并)
```

如果并行执行：
```
线程1: 判断 A+B → 结果=X (但此时不知道B是否已经与C合并)
线程2: 判断 B+C → 结果=Y (但此时不知道A是否已经与B合并)
线程3: 判断 C+D → 结果=Z
```

### 2.2 如果按批次并行，每批内串行 (方案B)

**并行方式：** 每次仍然分析20对（1个API调用），但多批次之间并行

```
批次1 (索引0-19):  [文本对1-20] → API调用1 → 顺序处理结果1-20
                        ↓
批次2 (索引20-39): [文本对21-40] → API调用2 → 顺序处理结果21-40
                        ↓
批次3 (索引40-59): [文本对41-60] → API调用3 → 顺序处理结果41-60
```

这种方式是当前代码的并行化，**不会影响批次内部的合并效果**。

## 三、问题影响评估

### 3.1 方案A (完全并行) 的影响

**对合并效果的影响：**

1. **错误合并：** 应该合并的被分开，或不应该合并的被合并
   - 例如：A应与B合并，但并行判断时A和B被分别与不同块合并

2. **文本截断/丢失：** 
   - 合并时文本拼接顺序错误，导致语义不通

3. **映射关系错误：**
   - `original_blocks` 记录错误，后续翻译结果无法正确拆分

**具体案例：**

假设原文块序列：[A, B, C, D, E]，正确顺序执行后：
- 假设 A+B 合并 → current_merged = A+B
- B+C 不应合并 → current_merged = B
- C+D 合并 → current_merged = C+D
- D+E 不应合并 → current_merged = D

如果并行判断，可能出现：
- A+B → 不合并 (误判，假设是API判断错误)
- B+C → 合并 → 但此时B已经被使用

### 3.2 方案B (批次间并行) 的影响

**对合并效果的影响：** 较小

因为：
1. 每批次内仍然是顺序处理
2. 批次边界天然划分了"可能需要分开的点"
3. 不同批次之间的块本来就没有直接的依赖关系

**但是，仍有边界问题：**

```
批次1: 文本对 [0-1], [1-2], [2-3], ... [18-19]
       处理后: current_merged = 某个块组合

批次2: 文本对 [19-20], [20-21], ...
       第1对的"块19"可能属于批次1处理后的 current_merged
       如果直接使用 text_blocks[19] 的原始文本，可能与实际状态不一致
```

**解决方案：** 批次间并行需要传递"合并状态"

## 四、推荐解决方案

### 4.1 方案一：批次间并行 (推荐)

**核心思路：** 保持批次内顺序处理，多批次之间并行执行

**实现方案：**

```python
def merge_semantic_blocks_with_llm_parallel(text_blocks, semantic_analyzer, source_lang):
    """使用多线程并行进行LLM语义块合并

    关键设计：
    1. 将文本块分成多个批次，每批20对
    2. 多批次之间并行调用API
    3. 每批次内保持顺序处理
    4. 批次间通过"断点"状态传递实现衔接
    """

    # 第一步：分批，生成所有文本对
    text_pairs = []
    for i in range(1, len(text_blocks)):
        prev_text = text_blocks[i-1].block_text
        curr_text = text_blocks[i].block_text
        text_pairs.append((prev_text, curr_text))

    # 将文本对分成多个批次，每批20对
    batch_size = 20
    batches = [text_pairs[i:i+batch_size] for i in range(0, len(text_pairs), batch_size)]

    # 第二步：多线程并行调用API
    with ThreadPoolExecutor(max_workers=5) as executor:
        # 并行执行各批次的语义分析
        futures = [
            executor.submit(semantic_analyzer.batch_analyze_semantic_relationship, batch, source_lang)
            for batch in batches
        ]
        # 按顺序收集结果
        all_merge_results = []
        for future in futures:
            all_merge_results.extend(future.result())

    # 第三步：顺序处理所有结果（与原逻辑一致）
    # ... (保持原有的合并逻辑)
```

**优点：**
- 显著提升性能（5个批次并行，耗时减少约80%）
- 保持合并逻辑不变，合并效果不受影响
- 实现相对简单

**缺点：**
- 需要处理批次边界的断点问题
- 线程数量需要控制，避免API限流

### 4.2 方案二：两阶段合并 (更安全)

**核心思路：** 先让LLM判断所有文本对的合并关系，再根据判断结果进行合并

**实现方案：**

```python
def merge_semantic_blocks_two_phase(text_blocks, semantic_analyzer, source_lang):
    """两阶段合并方案

    阶段1: 并行让LLM判断所有文本对的语义关系
    阶段2: 根据所有判断结果，顺序执行合并

    优势：完全解耦API调用和合并逻辑
    """

    # ========== 阶段1: 批量获取所有合并判断 ==========
    text_pairs = []
    for i in range(1, len(text_blocks)):
        text_pairs.append((
            text_blocks[i-1].block_text,
            text_blocks[i].block_text
        ))

    # 并行调用API获取所有判断结果
    merge_decisions = parallel_batch_analyze(text_pairs, source_lang)
    # merge_decisions[i] = True/False 表示 text_pairs[i] 是否应合并

    # ========== 阶段2: 根据判断结果合并 ==========
    current_merged = MergedBlock(block_text=text_blocks[0].block_text, ...)

    for i in range(1, len(text_blocks)):
        should_merge = merge_decisions[i-1]  # 使用预存的判断结果

        # 后续逻辑保持不变...
        if should_merge:
            # 合并
            current_merged.block_text += " " + text_blocks[i].block_text
            current_merged.original_blocks.append(text_blocks[i])
        else:
            # 保存并新建
            merged_blocks.append(current_merged)
            current_merged = MergedBlock(...)

    return merged_blocks
```

**优点：**
- 彻底解耦，API调用和合并逻辑完全独立
- 合并效果完全保证与原逻辑一致
- 可控制并行度

**缺点：**
- 如果文本块很多（如1000+），需要大量并行调用
- 需要处理API调用失败的情况

### 4.3 方案三：滑动窗口并行 (最佳性能)

**核心思路：** 使用滑动窗口，将有依赖关系的"链"分组并行处理

```
文本块: A B C D E F G H I J K L M N O ...

分组策略：
- 窗口1: A B C D ... (处理完生成断点)
- 窗口2:     C D E F ... (以上一个窗口的断点为起点)
- 窗口3:         E F G H ...
```

**优点：**
- 最大化并行度
- 保持长依赖链的完整性

**缺点：**
- 实现复杂，需要管理窗口状态
- 窗口边界可能产生不一致
