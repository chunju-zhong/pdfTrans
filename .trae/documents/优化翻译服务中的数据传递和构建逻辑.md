# 优化翻译服务中的数据结构构建逻辑

## 问题分析

用户指出了一个重要问题：在PDF提取阶段已经只提取了需要翻译的页面，因此在翻译服务中不需要再判断文本块是否在目标页面范围内。

当前实现中存在以下问题：
1. **冗余的页面范围检查**：在多个地方重复检查文本块是否在目标页面范围内
2. **数据传递冗余**：同时传递了`merged_blocks/all_blocks`和`extracted_content`给处理函数
3. **提前创建空对象**：提前创建所有目标页面的空`PdfPage`对象，可能包含不需要的页面
4. **代码可读性**：参数传递过多，函数职责不够清晰

## 优化方案

### 方案：简化数据结构构建逻辑

1. **完全移除`create_page_translated_blocks_dict`函数**：
   - 不再提前创建空的页面级别字典
   - 而是在处理过程中动态创建`PdfPage`对象

2. **移除不必要的页面范围检查**：
   - 移除`process_merged_blocks`中的页面范围检查
   - 移除`create_page_translated_blocks_dict`中的页面范围检查

3. **修改处理函数**：
   - `process_merged_blocks`：移除`extracted_content`和`target_pages`参数，在处理过程中动态构建页面字典
   - `process_original_blocks`：移除`extracted_content`和`target_pages`参数，在处理过程中动态构建页面字典

4. **更新函数实现**：
   - 在处理函数内部初始化空的`page_translated_blocks_dict`
   - 当处理到新页面时，检查字典中是否存在该页面的`PdfPage`对象，如果不存在则创建
   - 将翻译后的文本块添加到对应的页面中

## 具体修改步骤

1. **移除`create_page_translated_blocks_dict`函数**

2. **修改`process_merged_blocks`函数**：
   - 更新函数签名，移除`extracted_content`和`target_pages`参数
   - 移除页面范围检查逻辑
   - 在函数开始初始化空的`page_translated_blocks_dict`
   - 在处理翻译块时，动态检查和创建页面对象

3. **修改`process_original_blocks`函数**：
   - 更新函数签名，移除`extracted_content`和`target_pages`参数
   - 在函数开始初始化空的`page_translated_blocks_dict`
   - 在处理翻译块时，动态检查和创建页面对象

4. **修改`process_translation`函数**：
   - 更新处理函数调用，移除传递的`extracted_content`和`target_pages`参数

5. **更新测试用例**：确保测试代码与新的函数签名匹配

## 预期效果

1. **减少冗余代码**：移除了不必要的页面范围检查和`create_page_translated_blocks_dict`函数
2. **提高代码可读性**：函数参数更加简洁，职责更加清晰
3. **优化内存使用**：只创建实际需要的`PdfPage`对象，避免创建空对象
4. **保持功能完整性**：优化不会影响任何现有功能

## 风险评估

- **低风险**：修改仅涉及内部实现，不影响外部API
- **兼容性**：需要更新相关测试用例，但不会影响其他模块
- **测试覆盖**：所有测试用例应该继续通过，确保功能完整性
