# 批量语义分析提示词详细修改方案

## 问题分析

当前提示词存在以下问题，导致文本对5被错误判断为不合并：

1. **语法延续性规则不完善**：没有覆盖普通句子的延续结构，特别是前一个块以名词结尾、后一个块以冠词开头的情况

2. **列表开始标记识别过于宽松**：可能将以冒号结尾的普通短语误识别为列表开始标记

3. **缺少多级延续支持**：没有考虑列表项的多级延续情况，即列表项延续的延续

4. **上下文理解不足**：没有充分考虑前面的列表项内容，缺乏语义上下文理解

## 修改方案

### 1. 增强语法延续性规则

#### 扩展常见短语延续
- **添加普通句子延续模式**：
  - 前一个块以名词结尾（如 "information"）+ 后一个块以冠词开头（如 "the"）
  - 前一个块以形容词结尾 + 后一个块以名词开头
  - 前一个块以动词结尾 + 后一个块以名词/代词开头

#### 增强语法结构检测
- **添加更广泛的语法结构**：
  - 前一个块以名词、形容词、动词结尾
  - 后一个块以冠词、形容词、名词、动词开头
  - 整体构成完整语法结构

### 2. 优化列表开始标记识别

#### 严格定义列表开始标记
- **修改列表开始标记识别规则**：
  - 必须是引出列表的短语（如 "including", "such as", "consisting of", "the following" 等）
  - 单独以冒号结尾的短语不一定是列表开始标记
  - 列表开始标记通常出现在列表项之前，而不是句子中间

#### 调整处理逻辑
- **修改列表开始标记处理规则**：
  - 只有当块2是列表项时，块1末尾的冒号才被视为列表开始标记
  - 否则，以冒号结尾的短语应视为普通句子的一部分

### 3. 添加多级延续支持

#### 新增多级延续识别规则
- **定义多级延续**：
  - 一级延续：列表项 → 非列表项（当前已支持）
  - 二级延续：非列表项 → 非列表项（新增）
  - 二级延续是一级延续的延续，应与一级延续合并

#### 修改判断逻辑
- **添加多级延续判断**：
  - 如果块1是列表项的延续（一级延续），块2是非列表项，则块2是二级延续
  - 二级延续应与块1合并

### 4. 增强上下文理解

#### 添加语义上下文分析
- **考虑前面的列表项**：
  - 分析当前文本对与前面列表项的语义关系
  - 如果当前文本对是前面列表项的语义延续，应合并

#### 优化逻辑句子判断
- **强化逻辑句子判断**：
  - 考虑更广泛的语义联系
  - 分析文本块的内容和结构，判断是否属于同一个逻辑句子
  - 对于技术文档，考虑专业术语和句式结构

### 5. 调整规则优先级

#### 重新定义规则优先级
1. **标题判断**（最高优先级）
2. **列表项判断**（第二优先级）
3. **多级延续判断**（新增，第三优先级）
4. **语法延续性判断**（第四优先级）
5. **逻辑句子判断**（最低优先级）

#### 修改具体规则
- **规则5**：修改为"如果块1是列表项或列表项的延续，块2是非列表项且是其延续，返回: true"
- **规则6**：增强语法延续性检测，包括普通句子延续
- **规则7**：强化逻辑句子判断，考虑更广泛的语义联系

### 6. 更新示例

#### 添加多级延续示例
- **示例4**：多级延续示例
  - 对1: 列表项 → 非列表项（一级延续）
  - 对2: 非列表项 → 非列表项（二级延续）
  - 正确输出：{"merge": [true, true]}

#### 添加普通句子延续示例
- **示例5**：普通句子延续示例
  - 对1: "existing knowledge and dynamically retrieved information" + "the 'evidence' for the agent's response:"
  - 正确输出：{"merge": [true]}
