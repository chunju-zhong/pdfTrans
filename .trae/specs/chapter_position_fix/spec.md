# 章节位置信息提取修复 - 产品需求文档

## Overview

* **Summary**: 修复PDF翻译工具中章节位置信息提取的问题，确保正确使用PyMuPDF的get_toc(simple=False)方法获取带位置信息的详细书签，从而实现基于位置数据的章节归属判断。

* **Purpose**: 解决当前章节位置信息提取不准确的问题，确保跨章节页的文本块能够根据其垂直位置正确归属到相应章节。

* **Target Users**: PDF翻译工具的用户，特别是需要按章节生成翻译结果的用户。

## Goals

* 正确提取PDF书签的位置信息（页码和坐标）
* 实现基于位置数据的章节归属判断
* 提高跨章节页的文本块归属准确性
* 保持与现有代码的兼容性

## Non-Goals (Out of Scope)

* 不修改现有的章节识别逻辑
* 不改变章节树的构建方式
* 不影响其他功能模块的正常运行

## Background & Context

当前的章节位置信息提取逻辑存在问题，导致位置数据没有被正确使用，从而使得文本块的章节归属仍然基于整页判断，而不是基于位置数据。

根据PyMuPDF的文档，`doc.get_toc(simple=False)`方法返回的结果是一个列表，列表中每个元素是长度为5的元组（PyMuPDF 1.20+版本），格式如下：
```
(层级, 书签标题, 页码, 目标类型, 目标参数)
```

其中，目标参数是核心位置数据，内部跳转时是字典，包含`to`（目标坐标）、`zoom`（缩放比例）等。

当前代码的问题在于：
1. 提取位置信息的逻辑不正确，没有正确处理PyMuPDF返回的元组结构
2. 位置信息没有被正确存储和使用

## Functional Requirements

* **FR-1**: 正确提取PDF书签的位置信息（页码和坐标）

* **FR-2**: 实现基于位置数据的章节归属判断

* **FR-3**: 提高跨章节页的文本块归属准确性

* **FR-4**: 保持与现有代码的兼容性

## Non-Functional Requirements

* **NFR-1**: 性能影响最小化，不显著增加提取时间

* **NFR-2**: 代码可读性和可维护性良好

* **NFR-3**: 向后兼容，不破坏现有功能

## Constraints

* **Technical**: 基于现有的PyMuPDF库和章节识别逻辑

* **Dependencies**: 依赖现有的ChapterIdentifier类和TextBlock类

## Assumptions

* PDF文档包含有效的书签信息，用于章节识别

* 章节标题在页面上的位置相对固定，通常位于页面顶部

* 文本块的垂直位置（y坐标）可以用来判断其章节归属

## Acceptance Criteria

### AC-1: 位置信息提取准确性

* **Given**: PDF文档包含书签信息

* **When**: 提取书签信息

* **Then**: 正确提取书签的位置信息（页码和坐标）

* **Verification**: `programmatic`

### AC-2: 基于位置的章节归属

* **Given**: PDF文档包含跨章节页

* **When**: 提取文本块并关联章节信息

* **Then**: 文本块应根据其垂直位置正确归属到相应章节

* **Verification**: `programmatic`

### AC-3: 性能影响最小化

* **Given**: 处理包含多个章节的PDF文档

* **When**: 提取文本块并关联章节信息

* **Then**: 处理时间不应显著增加（不超过10%）

* **Verification**: `programmatic`

### AC-4: 向后兼容性

* **Given**: 处理不包含跨章节页的PDF文档

* **When**: 提取文本块并关联章节信息

* **Then**: 章节归属结果应与之前一致

* **Verification**: `programmatic`

## Open Questions

* [ ] 如何处理不同版本的PyMuPDF返回的不同格式的书签信息？

  * **解决方案**：使用try-except结构兼容不同版本的返回格式

* [ ] 如何验证位置信息提取的正确性？

  * **解决方案**：编写测试用例，验证位置信息的提取和使用

* [ ] 如何处理没有位置信息的书签？

  * **解决方案**：对于没有位置信息的书签，使用原有的整页判断逻辑
