# PDF翻译工具项目规则

## 1. 项目概述
- **项目名称**：PDF翻译工具
- **项目目标**：创建一个支持多翻译API的PDF翻译工具，能够准确提取PDF文本，高质量翻译，并生成格式良好的新PDF
- **主要功能**：
  - PDF文本提取
  - 多翻译API支持（aiping/硅基流动）
  - PDF生成
  - Word生成
  - Web界面
- **项目地址**：https://gitee.com/chunju/pdfTrans

## 2. 技术栈规范
- **Python版本**：3.9+
- **虚拟环境**：conda
- **PDF处理**：PyMuPDF 1.23+，camelot-py[cv]，opencv-python
- **Web框架**：Flask 3.0+
- **翻译API**：aiping API，硅基流动API
- **文档生成**：python-docx（用于Word文档生成）
- **API客户端**：openai（用于调用翻译API）
- **测试框架**：pytest
- **依赖管理**：使用requirements.txt和environment.yml
- **版本控制**：Git + Gitee

## 3. 代码风格
- **Python风格**：严格遵循PEP 8规范
- **缩进**：4个空格，禁止使用制表符
- **行宽**：最大100字符
- **命名规范**：
  - 类名：大驼峰命名法（如：PdfExtractor）
  - 函数名：小驼峰命名法（如：extract_text）
  - 变量名：小驼峰命名法或下划线命名法（如：textContent或text_content）
  - 常量名：全大写，下划线分隔（如：MAX_TEXT_LENGTH）
- **注释**：
  - 类和函数必须有文档字符串
  - 复杂逻辑必须有单行注释
  - 使用英文注释
- **函数设计**：避免使用嵌套函数，注重代码的可读性、结构化和可维护性
- **对象访问**：优先使用对象属性访问语法（如：`block.block_text`），而不是字典访问语法（如：`block['block_text']`），提高代码的可读性和类型安全性
- **数据结构**：优先使用数据模型类（如：`TextBlock`、`MergedBlock`）来表示数据，避免直接使用字典对象传递数据，提高代码的可维护性和类型安全性

## 4. 代码结构

### 4.1 文件组织
- **文件大小**：单个Python文件应控制在500行以内，超过则考虑拆分
- **模块拆分**：根据功能职责拆分为多个模块，每个模块专注于一个特定领域
- **导入顺序**：按照标准库、第三方库、项目内部模块的顺序导入，不同类别之间用空行分隔
- **相对导入**：优先使用相对导入（如：`from .text_block import TextBlock`），提高代码的可移植性

### 4.2 模块结构
- **模块布局**：每个模块应按照以下顺序组织代码：
  1. 导入语句
  2. 常量定义
  3. 类定义
  4. 函数定义
  5. 主代码（如果需要）
- **类结构**：类定义应包含：
  1. 类文档字符串
  2. 类属性（如果需要）
  3. `__init__`方法
  4. 其他方法（按照逻辑顺序）
- **函数结构**：函数定义应包含：
  1. 函数文档字符串
  2. 参数校验（如果需要）
  3. 核心逻辑
  4. 返回值

### 4.3 代码布局
- **空行使用**：使用空行分隔不同逻辑块，提高代码的可读性
  - 类与类之间：2个空行
  - 函数与函数之间：2个空行
  - 函数内部逻辑块之间：1个空行
- **括号使用**：遵循Python的括号使用规范，避免不必要的括号
- **运算符对齐**：多行表达式中的运算符应对齐，提高代码的可读性
- **代码分组**：相关的代码应分组在一起，使用空行分隔不同的逻辑组

### 4.4 复杂度控制
- **函数长度**：单个函数应控制在50行以内，超过则考虑拆分为多个函数
- **循环嵌套**：循环嵌套不应超过3层，超过则考虑重构
- **条件分支**：复杂的条件分支应考虑使用策略模式或其他设计模式重构
- **代码重复**：避免代码重复，使用函数或类封装重复的逻辑

### 4.5 错误处理
- **异常捕获**：只捕获必要的异常，避免使用裸露的`except:`语句
- **异常处理**：捕获异常后应进行适当的处理，如记录日志、返回错误信息等
- **异常传递**：对于不需要处理的异常，应允许其向上传递
- **自定义异常**：对于特定的业务逻辑错误，应定义自定义异常类

## 6. 模块化设计规范

### 6.1 模块化设计原则
- **单一职责原则**：每个模块只负责一个特定的功能领域
- **低耦合高内聚**：模块内部高度相关，模块之间依赖关系清晰简单
- **接口清晰**：模块对外提供明确的接口，隐藏内部实现细节
- **可测试性**：模块设计便于单元测试和集成测试
- **可维护性**：便于后续修改和扩展

### 6.2 模块划分标准
- **models/**：数据模型层，负责数据结构定义和数据访问
- **services/**：业务逻辑层，负责核心业务流程实现
- **utils/**：工具函数层，提供通用辅助功能
- **modules/**：核心功能层，封装第三方API或复杂功能组件

### 6.3 模块化设计目标
- 代码结构更加清晰，职责分明
- 便于维护和扩展
- 提高代码的可读性和可测试性
- 符合模块化设计原则
- 各模块之间低耦合，高内聚

### 6.4 模块化开发实践
- **优先构建数据模型**：在开发新功能时，首先定义清晰的数据模型类，避免直接使用字典传递数据
- **数据模型设计**：数据模型应包含完整的属性定义、类型提示和必要的方法
- **模块化依赖**：服务层和模块层应依赖于数据模型，而不是直接依赖于具体实现
- **接口设计**：模块对外提供明确的接口，隐藏内部实现细节

## 7. 开发流程
- **分支管理**：
  - main：主分支，仅用于发布稳定版本
  - develop：开发分支，整合各功能分支
  - feature/xxx：功能分支，用于开发新功能
  - bugfix/xxx：bug修复分支
  - release/xxx：发布分支，用于准备发布

- **代码提交规范**：
  - 提交信息格式：`[类型] 简短描述`
  - 类型包括：feat（新功能）、fix（bug修复）、docs（文档）、style（代码风格）、refactor（重构）、test（测试）、chore（构建/工具）
  - 示例：`feat: 添加百度翻译API封装`
  - 每次提交应只包含一个功能或修复
  - 提交前必须运行测试

- **代码审查**：
  - 所有代码必须通过至少一人审查才能合并到develop分支
  - 使用Gitee的Pull Request功能进行代码审查
  - 审查重点：代码质量、功能完整性、安全性、测试覆盖

## 8. 测试规范
- **测试目录**：所有测试脚本必须放在`tests/`目录下，命名格式为`test_*.py`
- **临时测试文件**：开发过程中临时创建的测试脚本及数据必须放到`tests/tmp/`目录
- **AI辅助开发文件**：AI辅助开发中生成的代码/数据分析脚本和相关数据必须放到`/Users/chunju/work/pdfTrans/.trae/tmp/`目录
- **单元测试**：每个模块必须有对应的单元测试
- **集成测试**：测试模块之间的交互
- **测试覆盖率**：目标覆盖率不低于80%
- **测试框架**：使用pytest

- **环境配置**：使用环境变量和配置文件管理
- **依赖管理**：使用requirements.txt和environment.yml锁定依赖版本
- **部署流程**：

- **API密钥管理**：
  - 所有API密钥必须存储在.env文件中
  - .env文件必须添加到.gitignore
  - 不得将API密钥硬编码到代码中

- **输入验证**：
  - 所有用户输入必须进行验证和清洗

- **文件处理**：
  - 处理文件时使用安全的临时目录
  - 对上传文件进行病毒扫描（可选）

- **错误处理**：
  - 不得向用户泄露敏感错误信息
  - 使用统一的错误处理机制
