# PDF翻译工具项目需求文档

## 1. 项目概述

### 1.1 项目背景
随着全球化的发展，跨语言PDF文档翻译的需求日益增长。现有的PDF翻译工具要么功能单一，要么依赖单一翻译API，无法满足用户多样化的需求。本项目旨在开发一个支持多种翻译API的PDF翻译工具，提供高质量、灵活的PDF翻译服务。

### 1.2 项目目标
- 创建一个支持多翻译API的PDF翻译工具
- 实现PDF文本和表格的准确提取
- 支持多种翻译服务（百度、aiping、硅基流动）
- 生成格式良好的翻译后PDF文档
- 提供简洁易用的Web界面
- 支持conda虚拟环境管理

### 1.3 适用范围
本工具适用于需要将PDF文档从一种语言翻译为另一种语言的用户，尤其是需要高质量翻译结果和保留原始格式的场景。

## 2. 功能需求

### 2.1 核心功能

#### 2.1.1 PDF文本提取
- **需求ID**：FR-001
- **描述**：能够从PDF文档中提取文本和表格内容
- **优先级**：高
- **技术要求**：
  - 支持提取普通文本，保留位置信息
  - 支持提取表格内容
  - 支持提取指定页面文本
  - 支持获取PDF元数据

#### 2.1.2 多翻译API支持
- **需求ID**：FR-002
- **描述**：支持多种翻译API服务
- **优先级**：高
- **技术要求**：
  - 支持百度翻译API
  - 支持aiping翻译API
  - 支持硅基流动翻译API
  - 实现统一的翻译接口，便于扩展

#### 2.1.3 PDF生成
- **需求ID**：FR-003
- **描述**：基于原始PDF生成翻译后的PDF文档
- **优先级**：高
- **技术要求**：
  - 保留原始PDF的布局和格式
  - 支持绘制翻译后的文本
  - 支持处理表格内容

#### 2.1.4 Web界面
- **需求ID**：FR-004
- **描述**：提供简洁易用的Web界面
- **优先级**：中
- **技术要求**：
  - 文件上传功能
  - 翻译服务选择
  - 语言选择
  - 翻译进度显示
  - 结果下载

### 2.2 辅助功能

#### 2.2.1 错误处理
- **需求ID**：FR-005
- **描述**：提供友好的错误提示和处理机制
- **优先级**：中

#### 2.2.2 日志记录
- **需求ID**：FR-006
- **描述**：记录系统运行日志，便于调试和监控
- **优先级**：中

## 3. 非功能需求

### 3.1 性能需求
- **需求ID**：NFR-001
- **描述**：处理中等大小PDF文档（<50MB）的时间不超过5分钟
- **优先级**：中

### 3.2 可靠性需求
- **需求ID**：NFR-002
- **描述**：系统稳定运行，故障率低于5%
- **优先级**：高

### 3.3 安全性需求
- **需求ID**：NFR-003
- **描述**：保护用户上传的文件和API密钥安全
- **优先级**：高
- **技术要求**：
  - API密钥使用环境变量管理
  - 上传文件定期清理
  - 不存储用户敏感信息

### 3.4 可扩展性需求
- **需求ID**：NFR-004
- **描述**：系统设计便于扩展新的翻译API和功能
- **优先级**：中

### 3.5 易用性需求
- **需求ID**：NFR-005
- **描述**：Web界面简洁易用，操作流程清晰
- **优先级**：中

## 4. 技术选型

### 4.1 开发语言
- **语言**：Python 3.9+
- **理由**：丰富的PDF处理库和Web框架支持

### 4.2 虚拟环境
- **工具**：conda
- **理由**：便于依赖管理和环境隔离

### 4.3 PDF处理库
- **库名**：PyMuPDF (fitz)
- **版本**：1.23+
- **用途**：PDF文本提取和生成

- **库名**：pdfplumber
- **版本**：0.10+
- **用途**：表格提取

### 4.4 Web框架
- **框架**：Flask
- **版本**：3.0+
- **理由**：轻量级、易于扩展

### 4.5 翻译API
- **百度翻译API**：支持多种语言翻译
- **aiping 模型调用API**：提供其部署的模型调用API，支持调用aiping模型来进行翻译
- **硅基流动 模型调用API**：提供其部署的模型调用API，支持调用硅基流动模型来进行翻译

### 4.6 版本控制
- **工具**：Git + Gitee
- **理由**：分布式版本控制，便于协作开发

## 6. 开发流程

### 6.1 分支管理
- **main**：主分支，仅用于发布稳定版本
- **develop**：开发分支，整合各功能分支
- **feature/xxx**：功能分支，用于开发新功能
- **bugfix/xxx**：bug修复分支
- **release/xxx**：发布分支，用于准备发布

### 6.2 代码提交规范
- 提交信息格式：`[类型] 简短描述`
- 类型包括：feat（新功能）、fix（bug修复）、docs（文档）、style（代码风格）、refactor（重构）、test（测试）、chore（构建/工具）
- 示例：`feat: 添加百度翻译API封装`

### 6.3 测试规范
- 所有测试脚本必须放在`tests/`目录下，命名格式为`test_*.py`
- 使用pytest框架进行测试
- 单元测试覆盖率不低于80%

## 7. 部署规范

### 7.1 环境配置
- 使用环境变量和配置文件管理
- 依赖版本使用requirements.txt和environment.yml锁定

### 7.2 部署流程
1. 从main分支拉取最新代码
2. 创建并激活虚拟环境
3. 安装依赖
4. 配置环境变量
5. 启动应用

## 8. 安全规范

### 8.1 API密钥管理
- 所有API密钥必须存储在.env文件中
- .env文件必须添加到.gitignore
- 不得将API密钥硬编码到代码中

### 8.2 输入验证
- 所有用户输入必须进行验证和清洗
- 限制上传文件大小（最大10MB）
- 限制上传文件类型（仅支持PDF）

### 8.3 文件处理
- 处理文件时使用安全的临时目录
- 对上传文件进行病毒扫描（可选）
